# SgodAI Market Radar Technical Design

## 1. Architecture Principle

SgodAI Market Radar uses a **Core Engine + Agent Copilot** architecture.

```text
Core Engine = facts, data, state, scoring, window detection, persistence, delivery
Agent Copilot = explanation, summary, attribution, research, Q&A, coordination
```

The Agent layer is not the system kernel. The kernel is composed of testable services, storage, provider interfaces, scoring engines, report generation, notification delivery, and auditable position state transitions.

## 2. System Layers

```text

        ↓

        ↓

        ↓

        ↓

Market Data / Announcements / News / Financial Reports / Policy /
Industry Data / Commodity Prices / Overseas Market / Sentiment
```

## 3. Core Rules

1. Facts live in database, not in agent memory.
2. Signals are generated by scoring engines, not free-form agent reasoning.
3. Agents explain and compose; core services calculate and persist.
4. Agents call tools; they do not bypass services or databases.
5. Position window states must be reproducible and auditable.
6. The system must remain usable when LLM or Agent is disabled.
7. Every AI output must be grounded to events, evidence fields, source IDs, confidence, and risks.
8. Agent is a Copilot, not an autonomous trader.

## 4. Pipeline

```text
Fetch
→ Normalize
→ Entity Link
→ Event Classify
→ Signal Enrich
→ Dedup
→ Impact Score
→ Trend Score
→ Sentiment Score
→ Risk Score
→ Position Window Detect
→ Store
→ Compose Report
→ Notify
```

Each pipeline step should be idempotent where possible and should emit traceable run metadata:

- run_id
- provider_id
- source_id
- input_hash
- output_hash
- started_at
- finished_at
- error_message

## 5. Service Modules

### DataService

Responsibilities:

- Call enabled DataProvider implementations.
- Normalize raw records into internal objects.
- Preserve raw source payload references.
- Maintain provider health and refresh timestamps.

### EventService

Responsibilities:

- Create canonical Event records.
- Deduplicate exact and semantic duplicates.
- Link events to assets, sectors, and graph nodes.
- Persist evidence and source references.

### SignalService

Responsibilities:

- Compute Impact, Trend, Sentiment, and Risk scores.
- Attach scoring evidence.
- Persist Signal records and score versions.
- Provide signal queries for reports and alerts.

### PositionWindowEngine

Responsibilities:

- Evaluate asset state from signals, market metrics, sector metrics, and risk flags.
- Persist state transitions.
- Provide deterministic explanations.
- Record rule_version and evidence IDs.

### ReportService

Responsibilities:

- Compose deterministic report skeletons.
- Optionally call LLMProvider for grounded language generation.
- Store generated reports with event and signal references.
- Expose report retrieval and export APIs.

### NotificationService

Responsibilities:

- Manage EmailTarget and NotificationTarget records.
- Filter report or alert delivery by target configuration.
- Send messages through pluggable NotificationProvider implementations.
- Persist DeliveryLog records and retry failures.

### KnowledgeGraphService

Responsibilities:

- Manage graph nodes and edges.
- Link events to graph nodes.
- Resolve impact paths from event to sector, asset, overseas mapping, commodity, and policy nodes.
- Provide graph context for reports and Agent tools.

## 10. Position Window Logic

### Scores

- Momentum Score
- Volume Score
- Fundamental Score
- Catalyst Score
- Sentiment Score
- Risk Score
- Relative Strength Score
- Valuation Zone Score

### State Rules

| State | Typical Conditions |
| --- | --- |
| 不关注 | Weak relevance, low signal quality, or out of user scope |
| 观察 | Rising Impact or Trend score without confirmation |
| 左侧建仓观察 | Depressed valuation/price zone, risk stabilizing, early improvement evidence |
| 右侧增持观察 | Trend confirmation, volume expansion, sector score improvement, repeated catalysts |
| 持有跟踪 | Stable trend and fundamentals, manageable risk |
| 减持观察 | Valuation pressure, weakening trend, falling sentiment, catalyst exhaustion |
| 风险预警 | Fast-rising Risk score, regulatory issue, earnings risk, liquidity risk, major negative event |

### Audit Requirements

Every state change must persist:

- previous_state
- current_state
- triggered_by_event_ids
- triggered_by_signal_ids
- support_factors
- risk_factors
- watch_variables
- rule_version
- confidence

## 13. Storage Plan

### MVP

- SQLite for metadata, events, signals, reports, delivery logs, and position states.
- Local filesystem for raw payloads, report exports, and attachments.
- JSON fields for flexible metadata during early iteration.

### Future

- PostgreSQL for multi-user/team deployment.
- DuckDB for local research analytics.
- pgvector, FAISS, or Chroma for semantic search.
- Object storage or NAS for large attachments.

## 15. Apple-first Frontend

The first UI targets macOS and Safari-friendly workflows:

- Dense dashboard layout
- Native system fonts
- Light/dark mode through system preferences
- Keyboard and pointer friendly controls
- Tables, matrices, timelines, graphs, and score cards
- Local storage for demo UI preferences
- No dependency on remote assets

The demo uses mock data through an adapter. Real provider integration should replace the adapter without changing view components.

## 16. Security And Compliance

The system must not:

- Place trades automatically
- Output deterministic buy/sell instructions
- Promise returns
- Bypass data-source authorization
- Hard-code API keys

The system must:

- Use environment variables for secrets
- Preserve original source references
- Record AI inputs and output provenance
- Mark confidence and risk notes
- Support data-source/model/agent enable and disable switches
- Display investment advice disclaimers in reports
